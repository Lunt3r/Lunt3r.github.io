FIDO基本信息

架构

具体流程

  1. 通过应用π演算对协议及其需要满足的安全目标进行描述；

  2. Proverif对用户代码进行语法检测、类型检测；

  3. Proverif根据攻击者子句规则、协议子句规则及安全目标翻译规则，将应用π演算描述下的协议会话、安全目标翻译成Horn子句；

  4. Proverif根据Horn子句推理规则将所以子句结合推理，产生大量推导子句，然后根据消解算法进行处理（保证推理不陷入循环）；

  5. 如果推导出的子句违背某一安全性质，Proverif会收集相关子句，并通过推导树在Horn子句层产生一条违背安全目标的路径；

  6. 对违背安全目标的路径在π演算层进行攻击重构。

安全目标：机密性、认证性、观察等价性等

三、安全协议最小化假设分析方法

  • 一般化：

    • 描述：自然语言提炼安全协议流程、安全假设、安全目标
    • 建模：分析工具（语句）进行描述
    • 分析：自动化输出

不足：描述不准确、完整；建模不准确；场景不同安全假设、攻击者能力都不同；结果分析不规范

3.2、形式化描述方法
3.2.1、协议流程

  1. 协议阶段划分

  2. 协议实体划分

  3. 协议初始知识

  4. 协议操作

3.2.2、安全假设

将攻击者的能力按环境做区分，有强有弱

根据密码算法（是否有漏洞）、数据（实体是否易被攻击导致数据泄露）、信道（私密信道会不会变成公开信道：诚实实体间未作身份校验、下载了恶意软件导致信道被中间人等）、其他等信息的安全与否建不同的模型

3.2.3、安全目标

####### 认证性目标

认证分类原则

  1. 存活

  2. 弱认证

  3. 非单射认证

  4. 单射认证

####### 机密性目标

在主动攻击者和被动攻击者的情况下，不同数据要求不同的机密性

####### 隐私性目标

敏感数据隐私性：可以转化为主动攻击者环境下的机密性

用户行为隐私性：不可链接性指的是在协议的多次会话中，攻击者无法区分会话中的消息是否来源于同一角色，它属于隐私性的一种。不可链接性可以直接作为形式化的语言描述安全协议的目标。而其他类型的隐私性目标可以使用进程等价性来进行描述（可以转化为观测等价性）

3.3、形式化建模方法
3.3.1、Proverif的能力

协议流程的建模，协议流程包含协议实体、实体的初始知识和协议的数据操作等。协议每个实体的具体操作使用进程宏 ( process macro）来描述，其中，进程宏的参数，以及全局变量参数，是每个实体的初始知识。通过在预先定义的信道上输入或输出 (in/out）消息，可以实现不同实体之间的通信。通过预先定义的函数，如加解密、签名函数，实现数据操作。

安全假设的建模包括密码算法、信道、数据保护等。通过应用π-演算中的函数(function）和等式（equation)，可以对密码算法进行建模。通过应用π-演算中包含的公开信道（public channel）和私密信道(private channel)，可以对安全假设中公开与非公开、非对称信道（Proverif手册6.7.4）的信道进行建模。

应用π-演算提供的询问（query）语句可以描述安全协议的安全目标。其中，attacker 关键字可以用于建模机密性目标，event， inj-event 关键字可以用于建模非单射协议和单射协议等认证性目标，equivalence, differ 等语句可以用于建模隐私性目标。

####### 3.3.2、进程通信场景下的恶意实体信道建模

基于Proverif的公开信道和私密信道进行组合

3.3.3、不可链接性建模

在验证不可链接性时可以转化为观测等价性，Proverif可以验证在外部攻击者对观测下，协议是否满足观测等价性。对于某些场景需要观测恶意实体时，则需要对恶意实体进行建模，可基于主动攻击者对初始知识进行扩充、对公开信道进行缩减，从而实现恶意实体的建模

3.4、最小化安全假设的验证方案

协议在最少的安全假设下能够达到所需的安全目标：找出协议的固定安全假设和可变安全假设，通过改变输入的安全假设并多次运行Proverif即可

可变安全假设过多则会导致遍历很困难，因此提出最小化假设算法：带缓存多遍历

反调试分析

发表于 2020-04-22 分类于 Android系列

调试器状态检测

根据android的官方文档，如果调试一个APK，必须满足以下两个条件中的任何一个：
1 APK的AndroidManifest.xml文件中的Application标签包含android:debuggable="true"
2 /default.prop中的ro.debuggable=1
根据Android SDK提供的android.os.Debug.isDebuggerConnected()方法可以进行判断程序是否被调试器调试了。

因此，当程序被设定为禁止调试时通过修改android:debuggable="true"来进行反反调试就行不通了，此时只能进行修改boot.img等方法来进行反反调试了。

调试器端口检测

调试器远程调试时，会占用一些固定的端口号，如23946。因此可进行读取/proc/net/tcp进行查看，若发现23946端口正在被使用，则说明程序在被调试。

可以在启动调试程序时修改端口，如使用命令./android_server -p12345以及adb forward tcp:12345 tcp:12345进行反反调试。

dump内存法一

  • 启动调试服务并进行端口转发

  • pull出/system/lib/libdvm.so文件

模糊测试

发表于 2020-03-26 分类于 Fuzzing系列

Fuzzing的概念

模糊测试（Fuzzing），是一种软件测试技术，通过向目标系统提供非预期的自动或半自动的生成随机数据输入并监视异常结果来发现软件或计算机系统漏洞。

AFL概念

AFL(American Fuzzy Lop)是一款基于覆盖引导（Coverage-guided）的模糊测试工具，它通过记录输入样本的代码覆盖率，从而调整输入样本以提高覆盖率，增加发现漏洞的概率。

AFL工作流程

  1. 从源码编译程序时进行插桩，以记录代码覆盖率（Code Coverage）
  2. 选择一些输入文件，作为初始测试集加入输入队列（queue）
  3. 将队列中的文件按一定的策略进行“突变”
  4. 如果经过变异文件更新了覆盖范围，则将其保留添加到队列中
  5. 上述过程会一直循环进行，期间触发了crash的文件会被记录下来
